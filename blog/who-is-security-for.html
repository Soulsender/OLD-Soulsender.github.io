Who is Security For?

February 8, 2023

It's a very general question. Security means different things to different people.

For some, it's the ability to hide data from others. For others, it's ensuring the correct data is received and used. Security is a thing that has existed since the beginning of humanity - but in the last century it has changed into a less-physical thing, and more of a digital thing as bank accounts, confidential documents and notes all turn into bytes on some kind of drive.

Today, digital security is often as important as physical security; the two are interwoven and you can't have one without the other. There is a saying in hacker communities that goes along the lines of "physical presence to a machine is the worst kind of vulnerability," and this is true - it's much easier to break into a machine by physically connecting to it with an ethernet cable than it is to break into it when it's behind firewalls, input sanitizing, and other defensive mechanisms. But at the same time, to stop people from breaking into those same machines physically, the same systems must be used for things like cameras, sensors, and alerting.

Unfortunately, people can frequently be seen disregarding digital security as "not important" because they seem to think it does not directly influence the physical world. Here's an interesting fact however - the biggest money heist ever to have been committed in history wasn't stealing bags of money by breaking into a bank, or by exploiting a very rich person.

It was done with cryptocurrency.

More specifically, it was done by the infamous North Korean Lazarus group when they stole 620 Million Dollars. This shows that it can sometimes have an even bigger impact on the physical world than one might think.

Ok, well what even is a vulnerability?

A huge problem with digital security is that unlike physical security, it is much harder to stop vulnerabilities from being created, due to the very nature of how computers work. A "vulnerability" is a bug in a piece of software, that a hacker might be able to exploit to gain some kind of access. Computers work in a sometimes painfully literal way. They will do exactly what you tell them to, but only exactly what you tell them to do. This leaves gaps where the programmer must account for what the users might input, and how the computer should interact with these inputs.

Let's take a very common and critical real-world vulnerability as an example - an SQL injection. Basically, SQL injections work as a consequence of a user's input not being "sanitized."

You know how sometimes if you try and type in characters like "abc" into a box where you are supposed to enter a credit card number, the website will refuse to allow you to use that as a valid input because it wants a number and not letters? This is called "input sanitizing" and works because the website compares what you entered to what the programmer intended you to enter, and if it is wrong, then it will not allow you to submit that value.

SQL injections sort of work the same way. When a user attempts to log in on a website and enter their username and password, the website will query the database on the server it is being hosted on, and ask the database if this user is found. If it is, then the website will allow you to login. If it is not, then it will throw an error, and prevent you from logging in.

Now the website will query the database in a way like this:

UserPassword = "password123";

txtUserPassword = getRequestString(UserPassword);
txtSQL = "SELECT * FROM Users WHERE UserPassword = " + txtUserPassword;

In this case, everything in the txtSQL variable is a SQL database query, where the SQL code is in between the "". The website is asking for a UserPassword which it will submit to the database for querying.

UserPassword = "' OR 1=1 '";

txtUserPassword = getRequestString(UserPassword);
txtSQL = 'SELECT * FROM Users WHERE UserPassword = ' + txtUserPassword;

However, what if we input something that wasn't a password? What if we input ' OR 1=1 '? This statement will actually end the SQL query, and exit out of it, and this gives us the ability to execute any code we want instead of only what the website expects. This can be abused to actually make the database query and return everything, not only our user, but every user password on the website. This works because the database reads the statement as if this password is found, OR if 1 = 1. Of course, one does equal one, and so it evaluates to true, and bypasses the false it would have come to originally.

So really, a security vulnerability is just logic that hasn't been thought through by the programmer, which can cause an error that can be exploited to break out of a controlled environment. In the case of this example, this SQL injection vulnerability could be remediated by sanitizing the user's input, and only allowing alphanumerical values to be submitted.
